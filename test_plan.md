# AI News 项目测试体系建设规划

## 背景

本项目是一个 AI 新闻日报生成器，核心流程：

```
RSS 抓取 → 关键词筛选 → AI 摘要 → 渲染输出
```

当前问题：
- **无测试**：`npm test` 是占位符，无任何单元测试/集成测试
- **无量化指标**：筛选规则（`src/processors/filter.ts`）用关键词子串匹配，但不知道准确率
- **简历需求**：想通过可量化的测试数据来体现项目改进成果

---

## 目标

建立一套 **"可测量 → 可改进 → 可验证"** 的测试评估体系：

1. 核心函数有**单元测试**覆盖
2. 筛选规则有**标注数据集 + 评估脚本**可量化精确率/召回率
3. 任何规则优化都能产出**具体的数字指标**
4. 简历可以写："将内容筛选准确率从 XX% 提升至 YY%"

---

## 指标体系

| 指标 | 含义 | 简历里的说法 |
|------|------|-------------|
| **精确率 (Precision)** | 筛选出的文章中真正 AI 相关的比例 | "内容准确率" |
| **召回率 (Recall)** | 所有 AI 文章中被正确筛选出的比例 | "覆盖完整度" |
| **F1 Score** | 精确率与召回率的综合评分 | — |

标注标准：文章标题/描述是否与 AI、ML、LLM、自动驾驶、机器人等主题**直接相关**。

---

## 分步实施

### Phase 1：搭建测试框架

做的事情：
- 安装 vitest（轻量、原生支持 TypeScript）
- 为 `src/processors/filter.ts` 的三个函数写单元测试
- 更新 `package.json` 的 `test` 脚本

涉及文件：
- `package.json` — 添加依赖
- `src/__tests__/filter.test.ts` — 新建
- `src/processors/filter.ts` — 不变（被测对象）

产出：
- `npm test` 可以跑，有测试通过的绿标

---

### Phase 2：建立标注数据集

做的事情：
- 跑一次完整抓取，获得原始文章池
- 创建一个交互式标注脚本，逐篇标记"相关/不相关"
- 目标是标注 200-300 篇，覆盖 5 个 RSS 来源
- 过程中发现当前筛选规则的误报/漏报模式

涉及文件：
- `scripts/label.ts` — 新建
- `data/labeled-dataset.json` — 新建

产出：
- 一个经过人工标注的数据集
- 对当前筛选质量有直观感受

---

### Phase 3：基线评估

做的事情：
- 用评估脚本在标注集上运行当前筛选规则
- 记录基线精确率/召回率/F1
- 分析具体失败案例

涉及文件：
- `scripts/evaluate.ts` — 新建
- `data/labeled-dataset.json` — 已标注的数据

产出：
- 基线指标报告（例如：精确率 83%，召回率 91%）
- 问题清单（哪些关键词导致误报最多）

---

### Phase 4：迭代改进 + 量化验证

| 改进 | 做法 | 预期效果 |
|------|------|----------|
| 词边界匹配 | `"AI"` → `/\bAI\b/i`，避免匹配 train/explain 等 | 精确率 +3-5% |
| 负向词黑名单 | 排除常见的误报模式 | 精确率 +2-4% |
| 评分制筛选 | 按关键词权重打分，设阈值 | 精确率 +3-5% |
| 标题相似度去重 | 补充现有的链接去重 | 少量提升 |

**关键**：每次改进后立即跑 `scripts/evaluate.ts`，记录增量变化，**所有数字可复现**。

涉及文件：
- `src/processors/filter.ts` — 迭代修改
- `scripts/evaluate.ts` — 对比运行

---

### Phase 5：生成简历素材

最终产出类似这样的对比表：

```
基线 (关键词子串)    精确率 83%  召回率 91%  F1 87%
优化后 (评分制)       精确率 92%  召回率 85%  F1 88%
```

简历文案模板：
```
• 搭建 AI 新闻筛选评估体系，建立 200+ 条标注数据集与自动化测试流程
• 通过数据分析定位筛选规则缺陷，优化关键词匹配策略与评分机制，
  将内容筛选准确率从 XX% 提升至 YY%
• 引入 vitest 单元测试与回归评估脚本，确保规则变更持续可验证
```

## Phase 4 实际执行结果

### 迭代改进路线

```
                                    精确率      召回率       F1
基线 (原始关键词子串匹配)             86.8%      92.2%     89.4%
Step 1: +agent, +具身智能, +token    87.7%     100.0%     93.4%
Step 2: +词边界, +OpenAI             92.8%     100.0%     96.2%
Step 3: +排除模式                    98.5%     100.0%     99.2%
                                   ─────      ─────     ─────
                                   +11.7%      +7.8%     +9.8%
```

### 修改内容

| 文件 | 变更 |
|------|------|
| `src/processors/filter.ts` | 新增 `textMatchesKeyword()` 函数（英文关键词 `\b` 词边界正则，兼容复数 `s?`）；新增 `OpenAI`、`agent`、`具身智能`、`token` 关键词；新增 `exclusionPatterns` 排除误报模式 |

### 排除模式说明

| 模式 | 目标 | 命中文章 |
|------|------|----------|
| `/^\d点\d氪/` | 36Kr 新闻汇总格式 | 8点1氪(百度职级) |
| `/^氪星晚报/` | 36Kr 晚报汇总格式 | 氪星晚报(AI4S) |
| `/扫地机器人/` | 产品品类非 AI | 追觅造车 |
| `/正大机器人/` | 投资方公司名 | 擎天租 |

### 剩余误报

仅 1 篇 FP，无通用排除模式，建议接受：

> **任正非最遗憾的一件事** — 匹配 "AI"（"卡住全球AI命脉"），文章主体关于华为历史

### 被否决的方案

评分制降权（AI/机器人→权重1，阈值2）会导致召回率崩盘至 ~35%，不可行（35+ 篇 TP 仅匹配 "AI" 单个关键词）。

---

## 验证方式

1. 单元测试：`npx vitest run`
2. 评估基线：`npx tsx scripts/evaluate.ts --brief`（纯数字，供对比）
3. 评估完整报告：`npx tsx scripts/evaluate.ts`
4. 端到端：`npm run dev` 跑完整管线
